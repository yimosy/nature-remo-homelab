# Nature Remo Homelab

Nature Remo を**ローカル API**(クラウド非経由)で操作する自宅サーバー用 Web サービスです。
Docker が動くマシンなら何でもホストにでき(Raspberry Pi、Linux サーバー、Synology などの NAS)、スマホのブラウザからリモコン操作できます。

## 特徴

- **ローカル完結** — Nature Remo Local API(`http://<RemoのIP>/messages`)で赤外線信号を学習・送信。インターネット不要
- **スマホ最適化 UI** — 大きなボタン、ダークモード対応、ホーム画面追加対応
- **端末制限** — 使う端末(ブラウザ)を登録制にし、管理者が承認した端末だけ操作可能
- **信号学習** — 実際のリモコンを Remo に向けて押すだけでボタンを登録(自動受信)
- **監査ログ** — 誰がいつ何を操作したかを記録、日次ローテート付き

## 構成

- FastAPI(Python)+ SQLite、フロントは静的な HTML/JS(依存はコンテナ内で完結)
- シングルコンテナ。データは `./data/`(SQLite とログ)に永続化

## 動作要件

- Docker(Compose v2)が動くこと — Raspberry Pi(64bit OS)、Linux サーバー、各社 NAS など
- Nature Remo と**同じネットワーク**に接続されていること
- Remo の IP アドレスがわかること(ルーターの DHCP 一覧などで確認。**固定 IP の割り当てを強く推奨**)

## セットアップ

```bash
git clone https://github.com/yimosy/nature-remo-homelab.git
cd nature-remo-homelab

# 管理者トークンを設定(.env は git 管理外)
cp .env.example .env
# .env を編集して ADMIN_TOKEN に長いランダム文字列を設定
#   生成例: openssl rand -base64 24

mkdir -p data
docker compose up -d --build
```

スマホのブラウザで `http://<サーバーのIP>:8000` を開けば使えます。
ポート 8000 が他サービスと被る場合は `docker-compose.yml` の `ports` を `"18000:8000"` のように変更してください。

### Synology NAS の場合の補足

- **Container Manager**(旧 Docker)→ プロジェクト → 作成 でこのフォルダを指定するか、SSH で上記コマンドを実行します
- Mac/PC からのファイル転送に rsync を使う場合は、コントロールパネル → ファイルサービス → rsync で「rsync サービスを有効にする」が必要です(無効だと SSH は通っても rsync だけ Permission denied になります)
- バインドマウント元のフォルダが自動作成されないため、`mkdir -p data` を忘れずに

### Raspberry Pi の場合の補足

- ベースイメージ(`python:3.12-slim`)はマルチアーチ対応なので、64bit OS(Raspberry Pi OS 64bit / Ubuntu)ならそのままビルドできます

## 初期設定の流れ

1. **端末登録と管理者ログイン** — ブラウザで開くと端末登録画面が出るので、端末名を付けて登録します。
   登録画面・承認待ち画面の下部にある「**管理者の方はこちら**」を開き、`ADMIN_TOKEN` でログインすると、
   承認前でも全機能(自分の端末の承認を含む)が使えるようになります。
2. **端末の承認** — 設定 → 「端末の管理」で、家族のスマホなどを個別に承認します。
3. **Remo の登録** — 設定 → 「Nature Remo」で名前と IP アドレスを追加し、「接続確認」を押します。
4. **家電と信号の登録** — 設定 → 「家電と信号」で家電(例: 💡 リビング照明)を追加し、
   「信号を学習」→ 実際のリモコンを Remo 本体の正面 10〜30cm に向けてボタンを1回押すと自動で受信されます。
5. **操作** — 「リモコン」タブに部屋(Remo)ごとの家電ボタンが並びます。タップで送信されます。

## 端末制限のしくみ

| レイヤー | 仕組み | 設定 |
|---|---|---|
| 端末単位 | ブラウザごとに Cookie トークンを発行し、管理者が承認・拒否 | UI から操作 |
| ネットワーク単位(任意) | 許可 CIDR 以外からのアクセスを一律 403 | `ALLOW_CIDRS: "192.168.1.0/24"` |

- 承認された端末のみ信号送信が可能。学習・登録・端末管理は管理者のみ。
- 端末を「拒否」にすると即座に使えなくなります。
- Cookie を消すと再登録(再承認)が必要になります。

## ログ

アクセスと操作の記録が `data/logs/access.log` に出力されます(アプリ内蔵のローテートで日次分割、`LOG_RETENTION_DAYS` 日分保持)。

- 形式: `UTC時刻 <TAB> 接続元IP <TAB> 操作者 <TAB> 内容` のタブ区切り
- API アクセス(メソッド・パス・ステータス)に加え、端末登録・承認、管理者ログインの成否、Remo/家電/信号の変更、信号送信の成否などの操作イベントを記録します

```
2026-07-04T09:20:24Z	192.168.1.21	父のiPhone(approved)	信号送信: リビング照明/点灯 -> 192.168.1.30
2026-07-04T09:21:02Z	192.168.1.99	-	管理者ログイン失敗
```

## 環境変数

環境変数は `docker-compose.yml` と同じフォルダの `.env` ファイルで設定します(`.env` は git 管理外)。

| 変数 | 必須 | 説明 |
|---|---|---|
| `ADMIN_TOKEN` | 必須 | 管理者ログイン用トークン。`.env` に設定 |
| `ALLOW_CIDRS` | 任意 | 許可するネットワーク(カンマ区切り CIDR)。未設定なら制限なし |
| `TRUST_PROXY` | 任意 | リバースプロキシ配下で `X-Forwarded-For` を信頼する場合に `true` |
| `LOG_RETENTION_DAYS` | 任意 | ログの保持日数(既定 30) |
| `DB_PATH` | 任意 | SQLite ファイルパス(既定 `/data/remo.db`) |

> **注意(ALLOW_CIDRS と Docker):** ブリッジネットワークでは接続元 IP が Docker のゲートウェイ IP に見えることがあります。その場合はリバースプロキシを前段に置いて `TRUST_PROXY: "true"` にするか、コンテナを host ネットワークで動かしてください。端末単位の承認制はこの影響を受けません。

## セキュリティ上の注意

- 本サービスは **LAN 内での利用を前提** としています。ホストのポートをインターネットに直接公開しないでください。
- 外出先から使いたい場合は VPN(WireGuard、Tailscale など)経由を推奨します。
- 通信は HTTP です。LAN 外に出す場合は必ず HTTPS 化(リバースプロキシ)してください。
